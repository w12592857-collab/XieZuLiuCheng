# AGENTS.md

本仓库是《动漫：开局顶上战争，暴打赤犬》的小说正文与写作流程仓库。任何 AI 或 Codex 接手续写、审稿、修补、整理资料时，必须先读取本文件，再读取总档案和当前章节上下文。

## 读取顺序

1. 先读 `AGENTS.md`，确认本项目硬规则。
2. 再读 `小说总档案.md`，确认正史、能力、伏笔、危机链和后续方向。
3. 写某一章前，必须读上一章正文、当前章章节卡、必要时读下一章规划。
4. 章节卡统一保存在 `ZhangJieKa` 文件夹，后续新增章节卡也必须放在这里。
5. 写正文前，必须读取 `RenWeiSuCaiKu` 文件夹内的微动作、口语、情绪反应和见证者反应材料，用来增强人味、画面感和去 AI 味。
6. 章节卡推荐命名为 `第XX章-章节标题-章节卡.md`，例如 `第25章-无惨震怒上弦空缺-章节卡.md`。
7. 如果发现总档案、章节正文、章节卡冲突，以当前 `XiaoShuo` 已写章节正文为最高正史，其次是小说总档案，再次是审查报告和总大纲。

## 项目定位

书名：《动漫：开局顶上战争，暴打赤犬》

类型：番茄男频爽文、动漫同人、诸天流、神级选择、名场面改写、暴打反派、拯救意难平。

核心卖点：主角林夜穿越诸天动漫世界，在关键名场面触发神级选择系统，专门改写原著里让读者憋屈的剧情。救艾斯，保宇智波，救炼狱，斩上弦，后续继续救自来也、五条悟等意难平角色。

这本书首先是爽文，不是慢热群像，不是设定说明书，不是压抑文学。每章都要让读者觉得主角在推进局面、打破憋屈、兑现爽点。

## 爽文硬规则

每章必须尽量满足以下链条：

危机出现 -> 危机来源明确 -> 系统选择或已有金手指介入 -> 奖励即时到账或能力合理调用 -> 金手指参与破局 -> 当章爽点兑现 -> 结尾追读钩子

如果一章只有铺垫，没有破局和爽点，必须压缩铺垫，把兑现提前。

如果连续两章都只压制敌人不解决敌人，第三章必须给明确结果，不能继续拖。

战斗不能水字数。推荐战斗写法：

危机压迫 -> 林夜一眼看穿 -> 一击破局 -> 反派破防 -> 结果落地

避免连续堆招式名、长段对砍、重复震惊。

## 系统与金手指规则

林夜拥有神级选择系统。每次系统给出选择后，林夜确认选择，对应奖励立刻发放，并且可以马上参与当章破局。

必须写清楚：

选择确认 -> 奖励到账 -> 奖励如何参与破局或埋伏笔

副本或阶段结束时仍可有结算奖励。即时奖励和阶段结算奖励不冲突，但不能混写成同一笔奖励。

金手指可以强，但不能无规则乱开。主角胜利必须来自已确立能力、即时到账奖励、剧情情报、提前准备、观察判断、敌人弱点或规则利用。

当前核心能力包括：太阳神火焰、三色霸气、剧情情报、永恒万花筒写轮眼、须佐能乎初始形态、日之呼吸、斩鬼之刃、世界锚点等。具体以 `小说总档案.md` 为准。

## 能力边界

太阳神火焰不属于恶魔果实体系，也不属于查克拉体系，可用于解释跨体系压制，但不能写成免疫所有规则。

见闻色能感知杀意、异常气息、空间波动、传谣路线、危险轨迹，但不能凭空知道没观察到的私密对话。

霸王色能压制普通人、传信者、低阶敌人，但不能替代政治处理、舆论修复和复杂心理博弈。

永恒万花筒和须佐能乎强，但不能把火影、鬼灭等世界的所有危机都一拳清空。环境、人质、规则、远程操盘和世界线反扑要继续制造压力。

## 知情范围

严格控制角色知道什么。

林夜可以知道原著剧情和系统提示，但不能把所有真相一次性告诉所有人。

配角只能知道自己亲眼见到、被告知、合理推理出的内容。

反派不能越界知道私密对话。黑绝、带土、无惨等可以通过监视、气息、血脉、情报网推断，但不能像作者旁白一样全知。

村民、路人、普通队员只能知道公开信息和碎片流言。

知情范围错误属于 P1 级硬伤，必须优先修。

## 第四面墙

第四面墙必须保持完整。

允许林夜作为穿越者在内心使用“原剧情、名场面、系统选择”等概念。允许系统提示任务与奖励。

禁止旁白跳出来说“读者、爽点、章节、作者、大纲、平台数据”等词。写作管理术语只能出现在流程文档和审稿报告里，不能进入小说正文。

## 每章生成流程

固定流程：

1. 总档案：确认正史、能力、人物状态、伏笔和下一章方向。
2. 章节卡：写清定位、承接、危机、金手指、爽点、伏笔、知情范围、结尾钩子。
3. 正文提示词：把章节卡转成可执行写作指令。
4. 正文：只输出小说正文，纯 txt 格式时第一行写章节标题。
5. 审稿：按 P1/P2/P3 检查剧情、逻辑、爽点、知情范围、时间线和画面感。
6. 修改：先修 P1，再修 P2，最后处理 P3。
7. 更新记录：更新总档案或连续性记录，包括能力、奖励、伏笔、反派状态和下一章钩子。
8. 下一章章节卡：为下一章准备清晰蓝图，保存到 `ZhangJieKa`。

## 字数与格式

用户要求计数时，每章第一行标题不计入字数。

如果用户要求“不超过 5000 字”，默认指从第二行正文开始统计非空白字符不超过 5000。

正文 txt 不加 Markdown 标题层级，不留行间空行，除非用户明确要求 Markdown。

章节卡、流程文档、审查报告可以使用 Markdown。

## 画面感要求

正文必须有可视画面，不要只写解释。

写正文前优先从 `RenWeiSuCaiKu` 中选择少量素材，结合当前角色和场景改写，不能机械照抄堆砌。

画面感包含：

- 人物动作
- 微表情
- 微动作
- 环境变化
- 对话
- 心理波动
- 场面反应
- 情绪起伏

每个关键爽点最好有见证者反应，让读者感到“当众打脸”“全场震动”“原著人物被救后情绪释放”。

## 审稿分级

P1 必须修：时间线错误、能力规则冲突、知情范围越界、系统奖励口径错误、核心因果断裂、第四面墙破裂。

P2 建议修：爽点不够、危机弱、伏笔无推进、反派降智、节奏拖、战斗水字数、画面感不足。

P3 可选修：错别字、标题格式、局部表达不顺、个别句子太像 AI。

## 禁止事项

不要擅自删除章节正文、总档案、章节卡、审查报告。

不要为了省事跳过章节卡和审稿。

不要让主角临时长出未铺垫的新大招。

不要让反派无理由降智。

不要让配角只会震惊，也要承担救人、牵制、判断、情绪反馈或见证功能。

不要上传 `.env`、API Key、私密项目数据或外层写作台代码。本仓库只管理小说正文和写作流程资料。
